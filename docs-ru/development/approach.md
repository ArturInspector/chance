## 1. Формализация задачи «Создать лендинг»

### 1.1 Цель G

```
G = "Создать лендинг проекта «Шанс» на Next.js"

measurable(G) = true
  ↳ Критерий: сайт доступен по localhost:3000, отображает все секции
  
controllable(G) = true
  ↳ Вся разработка в наших руках, нет внешних зависимостей
  
finite(G) = true
  ↳ Горизонт: MVP за 1-2 недели
```

### 1.2 Декомпозиция цели

Применяем алгоритм из раздела 2.1 `math.md`:

```
G: Создать лендинг
  ├─ G₁: Формализовать подход (этот документ)
  ├─ G₂: Инициализировать Next.js проект
  ├─ G₃: Создать главную страницу
  ├─ G₄: Добавить математическое обоснование
  ├─ G₅: Реализовать примеры (TikTok, Lisp)
  ├─ G₆: Добавить визуализацию графов
  └─ G₇: Деплой на Vercel
```

### 1.3 Граф зависимостей (DAG)

```
G₁ (подход) → G₂ (Next.js) → G₃ (главная) → G₄ (математика) → G₅ (примеры)
                                 ↓
                               G₆ (графы) → G₇ (деплой)
```

## 3. Итеративная разработка

### 3.1 Принцип микрошагов

Каждый компонент — атомарное действие. Не пишем сразу весь сайт.

```
Итерация 1: Философский фундамент (1 компонент)
Итерация 2: Алгоритм декомпозиции (1 компонент)
Итерация 3: Пример TikTok (1 компонент)
Итерация 4: Пример Lisp (1 компонент)
Итерация 5: Граф зависимостей (1 компонент)
```

### 3.2 Критерии готовности итерации

```
iteration_done(I) ⟺
  (1) Компонент отображается
  (2) Контент корректен
  (3) Нет ошибок в консоли
  (4) Git commit сделан
```

### 3.3 Процедура ежедневной работы

```
function daily_work():
  1. Открыть TODO list
  2. Выбрать MES с наивысшим приоритетом
  3. Выполнить MES (засечь время)
  4. Зафиксировать результат (commit)
  5. Обновить TODO list
  6. Если осталось время → вернуться к п.2
```

## 4. Процедурная спецификация компонентов

### 4.1 Философия компонентов

Каждый компонент React — процедура с:
- **Input** (props): preconditions
- **Output** (JSX): postconditions
- **Side effects**: взаимодействия

### 4.2 Структура компонента

```typescript
// Процедура: Отобразить философский фундамент
function PhilosophicalFoundation(props: Props) {
  // Preconditions
  assert(props.content !== null)
  
  // Атомарное действие: рендер
  return (
    <section>
      <h2>{props.title}</h2>
      <p>{props.content}</p>
    </section>
  )
  
  // Postconditions: секция отображена
}
```

### 4.3 Граф компонентов

```
App
  ├─ HeroSection (философский фундамент)
  ├─ ProblemSection (человек без спецификации)
  ├─ SolutionSection (декомпозиция)
  ├─ MathSection (алгоритмы)
  ├─ ExamplesSection
  │   ├─ TikTokProcedure
  │   └─ LispAnalogy
  ├─ GraphSection
  │   └─ DependencyGraph
  └─ CTASection (призыв к действию)
```

## 5. Рекурсивная проверка процедурности

### 5.1 Вопросы для каждого блока

Для каждого компонента/функции задаём:

1. **Что именно делаешь руками?**
   → Пишу JSX компонент с конкретным контентом

2. **Какие preconditions?**
   → Props, зависимости, контент из `math.md`

3. **Какие postconditions?**
   → Компонент отображается, контент корректен

4. **Сколько времени?**
   → Оценка duration для каждого MES

5. **Как проверить выполнение?**
   → Визуальная проверка в браузере, тесты

### 5.2 Пример для компонента TikTok

Действие: Создать компонент TikTokProcedure

## 6. Детекция разрывов в разработке

### 6.1 Типичные breakpoints

**1. Clarity breakpoint**: Не понятно, что делать дальше
- **Решение**: Декомпозировать задачу на более мелкие шаги

**2. Time breakpoint**: Задача занимает больше часа
- **Решение**: Разбить на несколько MES по 30-45 мин

**3. Dependency breakpoint**: Нужна библиотека/API
- **Решение**: Сначала MES на установку зависимости

**4. Energy breakpoint**: Слишком сложная задача
- **Решение**: Упростить или отложить

### 6.2 Алгоритм обнаружения

```
if застрял_на_задаче > 15 минут:
  → Декомпозировать дальше
  
if задача_занимает > 1 час:
  → Разбить на 2+ MES
  
if не_понятны_preconditions:
  → Изучить зависимости (отдельный MES)
```

## 7. Главная теорема разработки

**Теорема**: Если разработка формализована как граф атомарных MES, то проект завершится за конечное время.

**Доказательство**:
1. Граф конечен (MVP имеет ограниченный scope)
2. Граф ациклический (нет циклических зависимостей)
3. Каждый MES выполним (duration < 2 часов, controllable)
4. Последовательное выполнение MES уменьшает граф на 1 узел

→ Процесс терминирует за N шагов, где N = |MES|

∎

## 8. Метрики прогресса

### 8.1 Ежедневная метрика

```
progress_today = (completed_MES / total_MES) * 100%
```

### 8.2 Velocity

```
velocity = completed_MES / days_elapsed
```

Калибруем оценки: если actual_duration > estimated_duration, корректируем будущие оценки.

### 8.3 Оценка завершения

```
days_to_completion = remaining_MES / velocity
```
