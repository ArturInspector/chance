# Метаправила для формулирования правил

## Суть

Правила для создания правил любого проекта.  
Каждое метаправило применимо к документации, коду, процессам.

---

## 1. Операциональность

```
∀ rule R: ∃ verification_procedure(R)
```

Правило должно быть **проверяемо** через конкретную процедуру.

❌ "Код должен быть читаемым"  
✅ `lines(function) < 50 ∨ decompose(function)`

**Тест**: Можешь ли написать `check(rule) → true/false`?

---

## 2. Атомарность

```
∀ rule R: describes_one_concern(R)
```

Одно правило = одна проверка. Не смешивай constraints.

❌ "Функция короткая, хорошо названа, без side effects"  
✅ Три отдельных правила:
  - `lines(f) < 50`
  - `name_describes_action(f)`
  - `side_effects(f) = ∅`

**Тест**: Можно ли разбить правило на независимые части?

---

## 3. Приоритеты

```
∀ project: ∃ partial_order(rules)
```

При конфликтах высокоприоритетное побеждает.

| Уровень | Тип | Пример |
|---------|-----|--------|
| 0 | Критичный | Безопасность, корректность |
| 1 | Структурный | Атомарность коммитов, DAG |
| 2 | Стилевой | Именование, форматирование |
| 3 | Опциональный | Размер файла (можно нарушить) |

---

## 4. Минимальность

```
∀ rule R ∈ Rules: necessary(R) ∨ remove(R)
```

Убирай избыточные правила. Меньше правил = меньше конфликтов.

**Алгоритм**: Удали правило → качество упало? → вернуть. Нет? → удалить.

---

## 5. Эволюция

```
∀ rule R: measure(adherence(R)) → update(R) ∨ remove(R)
```

Правила живые. Если нарушается >30% времени → пересмотреть.

**Еженедельно**: Список нарушений → причина → изменить правило или автоматизировать.

---

## Чеклист нового правила

```
[ ] Операционально (есть verification procedure)
[ ] Атомарно (один concern)
[ ] Имеет приоритет (уровень 0-3)
[ ] Необходимо (без него качество падает)
[ ] Измеримо (можно отследить adherence)
```

---

**Главное**: Правило, которое нельзя проверить — не правило, а пожелание.
